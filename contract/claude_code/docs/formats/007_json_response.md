# Format: JSON Response

### Scope

- **Purpose**: Document the complete JSON response schema emitted by `claude -p --output-format json` on stdout.
- **Responsibility**: Single source of truth for the JSON response structure that `clr` must parse when implementing runner-level output transformations.
- **In Scope**: Top-level message fields, content block types and their fields, usage object structure, all nested objects.
- **Out of Scope**: Session JSONL format (-> `../jsonl/`); `stream-json` chunked format; `text` format (plain text, no structure).

### Location

- **Output**: stdout (not a file on disk)
- **Trigger**: `claude -p "<message>" --output-format json`
- **Encoding**: Single JSON object (not newline-delimited; contrast with `stream-json`)

### Schema

#### Top-Level Message Object

| Field | Type | Always Present | Description |
|-------|------|:-:|-------------|
| `id` | string | yes | API message ID (`msg_01ABC...`) |
| `type` | string | yes | Always `"message"` |
| `role` | string | yes | Always `"assistant"` |
| `model` | string | yes | Model ID used (e.g. `claude-sonnet-4-5-20250929`) |
| `stop_reason` | string \| null | yes | `"end_turn"`, `"stop_sequence"`, or `"max_tokens"` |
| `stop_sequence` | string \| null | yes | The matched stop sequence, or `null` |
| `content` | array | yes | Array of content blocks (see below) |
| `usage` | object | yes | Token usage statistics (see below) |

#### Content Blocks

The `content` array contains heterogeneous blocks distinguished by `type`.

##### Text Block

| Field | Type | Description |
|-------|------|-------------|
| `type` | string | `"text"` |
| `text` | string | The response text |

##### Thinking Block

| Field | Type | Description |
|-------|------|-------------|
| `type` | string | `"thinking"` |
| `thinking` | string | Extended thinking text |
| `signature` | string | Thinking block cryptographic signature |

##### Tool Use Block

| Field | Type | Description |
|-------|------|-------------|
| `type` | string | `"tool_use"` |
| `id` | string | Tool invocation ID (`toolu_01ABC...`) |
| `name` | string | Tool name (`Bash`, `Read`, `Edit`, etc.) |
| `input` | object | Tool-specific input fields |

##### Tool Result Block

| Field | Type | Description |
|-------|------|-------------|
| `type` | string | `"tool_result"` |
| `tool_use_id` | string | Back-reference to `tool_use.id` |
| `content` | string | Tool execution output |
| `is_error` | bool | Whether the tool execution failed |

#### Content Block Type Summary

| Block Type | Fields | Typical Presence |
|------------|--------|------------------|
| `text` | `type`, `text` | Always (at least one) |
| `thinking` | `type`, `thinking`, `signature` | When extended thinking enabled |
| `tool_use` | `type`, `id`, `name`, `input` | When tools were invoked |
| `tool_result` | `type`, `tool_use_id`, `content`, `is_error` | After tool_use execution |

#### Usage Object

| Field | Type | Always Present | Description |
|-------|------|:-:|-------------|
| `input_tokens` | u64 | yes | Input tokens consumed (uncached) |
| `output_tokens` | u64 | yes | Tokens generated by the assistant |
| `service_tier` | string | no | API service tier (`"standard"`) |
| `cache_creation_input_tokens` | u64 | no | Tokens used to create cache entries |
| `cache_read_input_tokens` | u64 | no | Tokens read from existing cache |
| `cache_creation` | object | no | Ephemeral cache breakdown (see below) |

##### cache_creation Sub-Object

| Field | Type | Description |
|-------|------|-------------|
| `ephemeral_5m_input_tokens` | u64 | 5-minute ephemeral cache tier tokens |
| `ephemeral_1h_input_tokens` | u64 | 1-hour ephemeral cache tier tokens |

### Example: Minimal Response

```json
{
  "id": "msg_01AEieWYMdbGML9PEKCmB36v",
  "type": "message",
  "role": "assistant",
  "content": [
    {"type": "text", "text": "Hello world"}
  ],
  "model": "claude-haiku-4-5-20251001",
  "stop_reason": "end_turn",
  "stop_sequence": null,
  "usage": {
    "input_tokens": 10,
    "output_tokens": 5
  }
}
```

### Example: Multi-Block Response (Thinking + Tool Use + Text)

```json
{
  "id": "msg_01XYZ...",
  "type": "message",
  "role": "assistant",
  "content": [
    {"type": "thinking", "thinking": "Let me check...", "signature": "sig_abc"},
    {"type": "tool_use", "id": "toolu_01DEF", "name": "Read", "input": {"file_path": "/tmp/test.rs"}},
    {"type": "tool_result", "tool_use_id": "toolu_01DEF", "content": "fn main() {}", "is_error": false},
    {"type": "text", "text": "The file contains a minimal Rust program."}
  ],
  "model": "claude-sonnet-4-5-20250929",
  "stop_reason": "end_turn",
  "stop_sequence": null,
  "usage": {
    "input_tokens": 120,
    "output_tokens": 85,
    "service_tier": "standard",
    "cache_creation_input_tokens": 0,
    "cache_read_input_tokens": 0,
    "cache_creation": {
      "ephemeral_5m_input_tokens": 0,
      "ephemeral_1h_input_tokens": 0
    }
  }
}
```

### Growth

Transient — not stored on disk. Each invocation produces one JSON object on stdout. Size scales with response length and number of tool use rounds (multi-block responses can reach 500KB+).
