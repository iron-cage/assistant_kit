# Verb: `run`

- **Kind:** canonical
- **Availability:** binary-only (library modules: exit 3)
- **`--dry-run`:** `cargo run -p <module> --bin <binary>` (binary modules only)

### Command

**Binary module:**
```bash
cargo run -p <module> --bin <binary> -- "$@"
```

Executes the module's primary binary entry point. `"$@"` forwards all additional arguments to the binary.

**Library module:**
```bash
echo "verb 'run' is not available for this project" >&2
exit 3
```

Exit code 3 signals "verb unavailable" per the `do` protocol. Tooling must distinguish exit 3 (not applicable) from exit 1 (tool failure) and exit 127 (tool not found).

### Notes

The `--bin <binary>` flag is required when a crate declares multiple binaries; it makes the entry point explicit. For single-binary crates it is still present for clarity and forward-compatibility.

Library modules exit 3 rather than omitting the script — the file must exist so that `verbs` can report `unavailable` and tooling does not treat a missing file as an error. The `verbs` output column shows `unavailable` for library modules.

`--dry-run` is only defined for binary modules — it prints the cargo invocation and exits 0. Library modules have no `--dry-run` branch (the script exits 3 unconditionally).

### Example

```bash
# claude_profile (binary: clp)
./verb/run                        # runs: cargo run -p claude_profile --bin clp
./verb/run --dry-run              # prints: cargo run -p claude_profile --bin clp
./verb/run -- --some-flag arg     # forwards args to clp binary

# claude_storage_core (library module)
./verb/run                        # exits 3; stderr: verb 'run' is not available for this project
```

`verbs` output comparison:
```
# binary module (claude_storage)
run      available   cargo run -p claude_storage --bin clg

# library module (claude_storage_core)
run      unavailable -
```

`verb/run` dispatcher — binary module (universal dispatcher body):
```bash
#!/usr/bin/env bash
# run — execute entry point; dispatches by VERB_LAYER to run.d/ layer.
set -euo pipefail
DIR="$(dirname "${BASH_SOURCE[0]}")/run.d"
LAYER="${VERB_LAYER:-}"
[[ -n "$LAYER" && -f "$DIR/$LAYER" ]] && exec "$DIR/$LAYER" "$@"
exec "$DIR/l1" "$@"
```

`verb/run.d/l1` — module-specific execution layer (default; example: `claude_runner`, binary: `clr`):
```bash
#!/usr/bin/env bash
# l1 — run clr binary directly (cargo ecosystem).
set -euo pipefail
SCRIPT_DIR="$( cd "$( dirname "${BASH_SOURCE[0]}" )" && pwd )"
cd "$SCRIPT_DIR/../.."
if [[ "${1:-}" == "--dry-run" ]]; then echo "cargo run -p claude_runner --bin clr"; exit 0; fi
exec cargo run -p claude_runner --bin clr -- "$@"
```

`verb/run` — library module (no dispatcher; unconditional exit 3):
```bash
#!/usr/bin/env bash
# do run — unavailable: library crate has no binary entry point
set -euo pipefail
echo "verb 'run' is not available for this project" >&2
exit 3
```

Each binary module's `run/runbox.yml` must declare `run_script: module/<name>/verb/run.d/l1` — the container entry point is the l1 layer directly. See `run/docs/parameter/015_run_script.md`. `verb/run` availability follows the same binary/library rule: `available` when the project has an entry point, `unavailable` (exit 3) when library-only.
