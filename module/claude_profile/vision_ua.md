# claude_profile — Візія

Claude Code має внутрішню файлову систему, яку ніхто не документує. Попрацювавши з ним достатньо довго, починаєш розуміти її структуру: розмови накопичуються в `~/.claude/projects/`, організовані за екранованим шляхом робочої директорії, облікові дані зберігаються в JSON-файлі певної форми, записи середовища сесій лежать у власній поддиректорії, статистика використання кешується поряд із налаштуваннями. Структура послідовна, відтворювана і цілком недокументована.

`claude_profile` перетворює цю неявну структуру на явний, типізований, протестований Rust API.
Це його головна мета, і все інше виходить з неї.

## Файлова система, якою він управляє

```
~/.claude/
  .credentials.json             ← активний OAuth-токен; форма — це API-контракт
  projects/
    -home-user-project-/        ← екранований шлях робочої директорії
      abc123.jsonl              ← історія розмов
      sessions-index.json
  session-env/{uuid}/           ← записи середовища для кожного виклику
  sessions/{id}.json            ← записи сесій
  stats-cache.json
  settings.json

$PRO/.persistent/claude/credential/   ← або $HOME/.persistent/claude/credential/
  work@acme.com.credentials.json  ← іменований знімок облікових даних
  personal@home.com.credentials.json
  _active                         ← «work@acme.com» ← один текстовий файл, одна відповідальність
```

Кожен шлях у цьому дереві, що має значення для інструментів, обчислюється через
[`ClaudePaths`](src/paths.rs) або [`PersistPaths`](src/persist.rs). Ніде в цьому коді немає рядкового літерала `~/.claude/` —
жодного разу. `ClaudePaths` — єдине авторитетне джерело, і кожен інший модуль,
якому потрібен шлях, звертається до нього.

## Головна проблема: ротація облікових записів

Підписки Claude Code обмежують п'ятигодинним вікном активного використання за цикл. Розробники
з кількома підписками — робоча, особиста, командна — змушені ротувати між ними, щоб зберігати
безперервну доступність. До появи цього крейту це означало відкрити текстовий редактор і
вручну замінити вміст `.credentials.json`. Одна несвоєчасна операція запису, одне
перерване збереження — і файл облікових даних напівзаписаний. Claude Code не запуститься.

Повне рішення виявилось єдиною атомарною файловою операцією.

```rust
// Повна реалізація перемикання облікових записів:
std::fs::copy( &src, &tmp )?;           // поетапно в сусідній .json.tmp
std::fs::rename( &tmp, &credentials )?; // атомарна заміна
std::fs::write( &marker, name )?;       // оновлення _active
```

Обидва файли знаходяться в `~/.claude/` — тобто на одній файловій системі, — що гарантує
атомарність rename. Збій у будь-який момент залишить або старі, або нові облікові дані —
але ніколи напівзаписаний файл. Це перетворює небезпечну ручну операцію на безпечний
однорядковий виклик:

```rust
let credential_store = claude_profile::PersistPaths::new()?.credential_store();
let paths            = claude_profile::ClaudePaths::new().expect( "HOME must be set" );
claude_profile::account::switch_account( "personal@home.com", &credential_store, &paths )?;
```

Збереження, перелік та видалення облікових записів завершують життєвий цикл. Збереження — це
копіювання. Видалення — це видалення з одним захистом: не можна видалити активний обліковий
запис. Захист не дає вказівнику облікових даних зависнути, бо інакше наступне перемикання
успішно завершиться, але буде вказувати в нікуди.

## Розрізнення, що має значення

Один аспект коректності, що потребував ретельного обмірковування: `expiresAt` у
`.credentials.json` — це строк дії **OAuth access token**, зазвичай кількома місяцями
попереду і автоматично оновлюється Claude Code до його закінчення. Це **не** п'ятигодинне
вікно використання підписки, яке є серверним станом і не має локального представлення.

`TokenStatus` тому класифікує саме OAuth-токен, а не вікно використання:

```rust
pub enum TokenStatus {
    Valid { expires_in: Duration },         // більше 60 хв залишилось
    ExpiringSoon { expires_in: Duration },  // менше 60 хв — варто ротувати
    Expired,                                // OAuth-токен минув expiresAt
}
```

Це розрізнення важливе для автоматизації. `Valid` токен не означає, що вікно підписки
ще не вичерпано — це окремий сигнал, що спостерігається лише при спробі запиту і отриманні
відповіді сервера. Помилка тут означала б автоматизацію, яка нібито працює (токен виглядає
коректно), але мовчки дає збій (вікно вичерпано). Правильне рішення — не робити
стверджень, які неможливо підтвердити.

## Межа, що є фічею

`claude_profile` НЕ ПОВИНЕН виконувати процеси. `std::process::Command` не імпортується
ніде в `src/`. Якщо він з'явиться — набір тестів миттєво завалиться:

```
RESPONSIBILITY VIOLATION: claude_profile MUST NOT import std::process::Command
```

Це не конвенція. Це тест статичного аналізу, що запускається при кожному коміті.

Чому це важливо? Тому що `claude_profile` зберігає шлях до облікових даних. Крейт, який
знає, де лежить `~/.claude/.credentials.json`, і при цьому запускає підпроцеси, створює
поверхню, що змішує доступ до даних із виконанням — такий тип поверхні, що, раз виникнувши,
має тенденцію розростатися. Суворе розділення означає, що код, який торкається облікових даних,
можна перевіряти в повній ізоляції. Крейт виконання (`claude_runner_core`) нічого не знає
про облікові дані. Крейт облікових даних нічого не знає про виконання.

## Поточний стан

Усі чотири бібліотечні модулі — `account`, `token`, `paths`, `persist` — реалізовані та повністю
протестовані. Бінарний шар CLI додає `adapter`, `output` і `commands` (10 команд за
фічею `enabled`). Функціональні вимоги знаходяться в [`docs/feature/`](docs/feature/) (FR-6 до FR-18);
кожен екземпляр документа пов'язаний з тестами через секції Cross-References. Усі тести проходять. Нуль попереджень clippy.

Крейт використовується в продакшні для визначення шляхів та ротації облікових записів.

## Однорядковий виклик (FR-13: Auto Rotate)

Кожен робочий процес ротації потребував одного й того самого шаблонного коду: список
облікових записів, фільтрація активного, вибір кандидата з найвищим `expires_at_ms`,
перемикання. Тепер ця логіка знаходиться в бібліотеці, а не у кожного клієнта:

```rust
// До (повторюється у кожного клієнта):
let candidate = account::list( &credential_store )?
    .into_iter()
    .filter( | a | !a.is_active )
    .max_by_key( | a | a.expires_at_ms )
    .ok_or_else( || /* немає доступних облікових записів */ )?;
account::switch_account( &candidate.name, &credential_store, &paths )?;

// Після (FR-13):
account::auto_rotate( &credential_store, &paths )?;  // повертає ім'я облікового запису, на який переключились
```

`auto_rotate()` — це FR-13, реалізована в `account.rs`. Вона завершує обіцянку
«безшовної ротації» і перетворює основний сценарій використання на однорядковий виклик.
