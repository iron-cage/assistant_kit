# dream — Візія

Claude Code — найпотужніший AI-асистент для розробки з доступних. При цьому в нього немає публічного API, немає програмного інтерфейсу, і немає жодного способу інтегрувати його в автоматизовані процеси — не пишучи парсер файлів і shell-виклики з нуля. Кожна команда, яка серйозно ним користується, врешті-решт пише один і той самий клей-код — або, що гірше, взагалі не пише і просто клацає мишею в терміналі вручну.

`dream` це змінює.

## Простір проблем

Коли починаєш серйозно працювати з Claude Code, натикаєшся на три чіткі точки тертя:

**1. Ротація облікових записів — ручна і ненадійна.**
Кожна підписка Claude має п'ятигодинне вікно активного використання. Якщо ти підтримуєш кілька підписок — робоча, особиста, командна — щоб не переривати роботу, то переключення між ними сьогодні означає ручне редагування `~/.claude/.credentials.json`. Якщо щось піде не так — вилітаєш із сесії повністю. Один збій запису — і твоєї сесії більше немає. Має бути кращий спосіб.

**2. Життєвий цикл сесій — непрозорий.**
Claude Code зберігає всю історію розмов у `~/.claude/projects/`, використовуючи неочевидний екранований шлях як ключ. Чи має новий запуск відновити існуючу розмову або почати нову? Де саме знаходиться історія для `/home/user/project`? Відповіді на ці питання зараз потребують ручного перегляду недокументованих структур директорій. Програмному коду, якому потрібно вирішити «продовжувати чи починати заново?» — нема що викликати.

**3. Автоматизація потребує саморобного клею.**
Будь-який інструмент, що хоче викликати Claude Code — CI-хук, бот для перевірки коду, завдання за розкладом — має знати, які прапорці передавати, як налаштувати середовище, як обробляти інтерактивний і неінтерактивний режими, як підключити stderr і stdout. Немає задокументованої угоди про виклики, крім довідки самого бінарного файлу.

## Що ми побудували

`dream` — це шаруватий Rust-workspace, що вирішує кожну з цих проблем:

```
claude_profile      Управління життєвим циклом облікових записів + повна топологія ~/.claude/
claude_runner_core  Типобезпечний builder-паттерн для виклику Claude Code
claude_runner       CLI-адаптер — людський інтерфейс над builder'ом
claude_storage_core JSONL-парсер для історії розмов
claude_storage      CLI для дослідження та аналізу сховища сесій Claude
```

**Ротація облікових записів — тепер два рядки:**
```rust
claude_profile::account::switch_account("work")?;
// ~/.credentials.json атомарно замінено, маркер _active оновлено
```

**Виявлення продовження — один виклик функції:**
```rust
if claude_profile::check_session_exists(&working_dir) {
    // відновити, не починати заново
}
```

**Виклик Claude Code — через builder:**
```rust
ClaudeCommand::new()
    .message("перевір цей PR на проблеми безпеки")
    .working_dir(&repo_path)
    .verbose(true)
    .execute()?;
```

## Архітектура

Крейти утворюють два чисті шари:

```
Шар 1 — Сховище та виявлення (без виконання процесів)
  claude_storage_core   JSONL-парсинг ~/.claude/, статистика токенів
  claude_profile        CRUD облікових записів, статус токенів, шляхи сесій
  claude_storage        CLI над storage_core

Шар 2 — Виконання (власник Command::new("claude"))
  claude_runner_core    Builder для ClaudeCommand, єдина точка виконання
  claude_runner         CLI-адаптер із YAML-схемою для інтеграції зі споживацьким робочим простором
```

Тверда межа між шарами забезпечується тестом статичного аналізу: якщо `std::process::Command` з'явиться в `claude_profile`, набір тестів одразу завалиться. Поділ відповідальності — не умовність, а інваріант, перевірений компілятором.

## Філософія проєктування

**Атомарні операції за замовчуванням.**
Переключення облікових записів використовує write-then-rename, а не перезапис на місці. Обидва файли знаходяться в одній директорії `~/.claude/` — тобто на одній файловій системі — що гарантує атомарність rename. Збій у будь-який момент залишить або старі, або нові облікові дані — але ніколи не напівзаписаний файл.

**Одне місце для кожного рішення.**
`ClaudePaths` — єдине авторитетне джерело всіх шляхів `~/.claude/` у всьому workspace. `ClaudeCommand::execute()` — єдина точка запуску процесів. Жодна з цих угод не є неформальною — обидві підкріплені тестами, які точно вкажуть, який файл порушив правило.

**Документація — джерело істини.**
Кожен крейт постачається з директорією `docs/`, що містить екземпляри doc-сутностей (feature, invariant, api, pattern, data_structure) із функціональними та нефункціональними вимогами, словником і архітектурою. Контрольний список — не прагнення, а вимога: кожен пункт має іменований тест, який повинен проходити, щоб галочка залишалася.

## Поточний стан

Workspace готовий до продуктивного використання на рівні L3 (nextest + doc tests + clippy, нуль попереджень). Усі п'ять крейтів проходять. Шар управління обліковими записами (`claude_profile` 0.2) нещодавно доданий і має повне тестове покриття всіх функціональних вимог.

Що перевірено в продуктиві:
- `claude_storage` парсить ~1 900 проєктів і 2 400 сесій (~7 ГБ JSONL) менш ніж за 30 секунд із теплим кешем
- Виявлення продовження сесій коректно працює в живих інструментах
- Builder `claude_runner` використовується для автоматизації багатокрокових ревʼю

Що специфіковано, але ще не перевірено в бою:
- Ротація облікових записів між живими підписками (реалізація завершена; цикл ротації в реальних умовах потребує часу)
- Моніторинг закінчення терміну дії токена як тригер автоматизації

## Відкриті завдання, варті уваги

Основа міцна. Цікава робота — попереду:

- **Автоматизація ротації**: демон або хук, що стежить за `TokenStatus` і автоматично ротує облікові записи, коли спрацьовує `ExpiringSoon` — та деталь, що робить роботу з кількома підписками справді безперервною.

- **Аналітика використання**: `claude_storage_core` вже парсить статистику токенів за сесіями. Інструмент, що показує витрати токенів за проєктом, за днем, за типом розмови, зробить управління підписками значно розумнішим.

- **Відтворення розмов**: JSONL-формат повністю розпарсений. Інструмент для вилучення, підсумовування або порівняння розмов між сесіями відкрив би принципово нові робочі процеси.

- **Інтеграція з CI**: `claude_runner`, що запускається на pull request'ах як структурований крок перевірки коду — з виводом, достатньо придатним для парсингу, щоб публікувати його як коментар.

- **Публікація на crates.io**: workspace спроєктований для цього з першого дня. Інваріант приватності вже забезпечено. Публікація `claude_profile` і `claude_runner_core` як публічних крейтів — логічний наступний крок після стабілізації інтерфейсів.

## Стек

Rust 2021, педантичний clippy із `missing_inline_in_public_items` і `std_instead_of_core` як жорсткими помилками. Кожен публічний елемент задокументований. Кожна функціональна вимога прив'язана до іменованого тесту.

Це код, який можна читати холодним поглядом і розуміти — тому що обмеження, що ускладнюють написання, — це ті самі обмеження, що полегшують підтримку.
